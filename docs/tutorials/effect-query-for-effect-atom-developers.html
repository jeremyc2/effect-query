effect-query for Effect Atom developers

If you already like thinking in atoms, this library feels very natural. You keep plain atoms for local state. You reach for query atoms when the value should be fetched, cached, refreshed, and hydrated for you.

The nicest part is that you do not need to abandon the atom mental model to get there. Server state does not become a separate universe.

The first shift is that remote state stops pretending to be local state

If you already think in atoms, effect-query is not asking you to relearn reactivity. It is asking you to stop hand-rolling server-state policy in places where plain atoms were never the right abstraction.

From the component's point of view, both values are atoms. The difference is that a query atom comes with cache identity, freshness, cancellation, and hydration already attached.

Effect Atom

Plain local state.

import * as Atom from "effect/unstable/reactivity/Atom";

const countAtom = Atom.make(0);

effect-query

Remote state with cache behavior attached.

import { createQueryAtom, makeRuntime } from "effect-query";

const queryRuntime = makeRuntime();

const dashboardAtom = createQueryAtom({
  runtime: queryRuntime,
  queryKey: ["dashboard"],
  queryFn: fetchDashboard,
});
In practice: keep plain atoms for local truth. Use a query atom when the value comes from somewhere else and should survive refetches, retries, and hydration.

Query functions get the request context plain atoms usually make you invent

Query atoms are still atom-shaped from the outside, but the underlying query function gets a little more structure than a plain async atom. It receives the resolved queryKey and an AbortSignal so request layers can handle cancellation.

That means you do not need to smuggle metadata through closures just to handle retries, logging, or abort-aware APIs. 

const taskAtomFactory = createQueryAtomFactory({
  runtime: queryRuntime,
  queryKey: (taskId: string) => ["task", taskId],
  queryFn: (taskId, { signal }) =>
    DemoApi.use((api) => api.getTask(taskId, { signal })),
});

Parameterized remote reads become query atom factories

Plain atoms are still great for local selection. The new move is that a parameterized remote read becomes a query atom factory. Call it with an argument and you get the atom you want to read.

This is the term worth learning early because it replaces a pile of ad-hoc "async atom with arguments" patterns with one consistent model.

Effect Atom

Local state chooses what is active.

const selectedUserIdAtom = Atom.make("42");

const greetingAtom = Atom.make((get) => {
  const userId = get(selectedUserIdAtom);
  return `Hello ${userId}`;
});

effect-query

The same selection can drive a remote cached read.

import { createQueryAtomFactory } from "effect-query";

const userAtomFactory = createQueryAtomFactory({
  runtime: queryRuntime,
  queryKey: (userId: string) => ["user", userId],
  queryFn: (userId: string) => fetchUser(userId),
});

const selectedUserIdAtom = Atom.make("42");

const selectedUserAtom = Atom.make((get) =>
  userAtomFactory(get(selectedUserIdAtom)),
);

Effect Atom

Reading an atom in React.

function Counter() {
  const count = useAtomValue(countAtom);
  return <span>{count}</span>;
}

effect-query

Reading a query atom feels almost the same.

import * as Cause from "effect/Cause";

function DashboardScreen() {
  const dashboard = useAtomValue(dashboardAtom);

  if (dashboard.isPending) return <Spinner />;
  if (dashboard.isError) {
    return <ErrorView error={Cause.pretty(dashboard.failureCause)} />;
  }
  if (dashboard.data !== undefined) {
    return <DashboardView dashboard={dashboard.data} />;
  }

  return null;
}
The trade-off: there is a little more vocabulary up front. The payoff is that parameterized remote state has one explicit home instead of being rebuilt from scratch in every feature.

Local writes stay atoms; remote writes become mutations

The division of labor is refreshingly simple. Local writes stay as writable atoms. Remote writes become mutations, because mutations know how to invalidate matching query atoms afterward.

This is one of the biggest quality-of-life improvements over "just use async atoms for everything." You stop making every write path rediscover cache invalidation the hard way.

Effect Atom

Local write.

const draftAtom = Atom.make("");

const saveDraftAtom = Atom.make((get, set, body: string) => {
  set(draftAtom, body.trim());
});

effect-query

Remote write.

import { createMutationAtom } from "effect-query";

const addCommentMutation = createMutationAtom({
  runtime: queryRuntime,
  mutationFn: (input: { taskId: string; body: string }) =>
    DemoApi.use((api) => api.addComment(input)),
  invalidate: (input) => ({
    task: [input.taskId],
  }),
});

Effect Atom

Imperative registry writes.

const registry = AtomRegistry.make();

registry.set(draftAtom, "hello");
const draft = registry.get(draftAtom);

effect-query

Imperative cache writes on the factory.

await Effect.runPromise(
  taskAtomFactory.setData("task-1", (current) => {
    const base = Option.getOrElse(current, () => emptyTask("task-1"));
    return {
      ...base,
      title: "Optimistic title",
    };
  }),
);

const cachedTask = await Effect.runPromise(taskAtomFactory.peek("task-1"));

Effect Atom

You usually own cancellation yourself.

// Plain async atoms do not come with a query cancellation helper.

effect-query

Query work can be cancelled through the factory or atom.

await Effect.runPromise(taskAtomFactory.cancel("task-1"));
The payoff: local state and remote state still feel related, but they stop pretending to be the same problem.

Yes, you can model remote data with plain atoms. That does not make it a good idea.

Yes, you can model remote data with plain atoms. The catch is that you then inherit all the boring parts: cache identity, freshness, retries, invalidation, hydration, and imperative cache helpers. Query atoms exist to take that burden off your plate.

Plain async atoms are still great when you truly want one-off async computation. They are a bad bargain when you actually want a shared server-state layer.

Effect Atom

It works, but you own the server-state details now.

const userAtom = Atom.make((get) =>
  Effect.promise(() =>
    fetch(`/api/users/${get(userIdAtom)}`).then((r) => r.json()),
  ),
);

effect-query

The runtime and factory take care of the extra machinery.

const userAtomFactory = createQueryAtomFactory({
  runtime: queryRuntime,
  queryKey: (userId: string) => ["user", userId],
  staleTime: "1 hour",
  gcTime: "5 minutes",
  refetchInterval: false,
  networkMode: "online",
  reactivityKeys: (userId: string) => ({
    user: [userId],
  }),
  queryFn: (userId: string) => fetchUser(userId),
});

Effect Atom

Plain atoms usually do not need shared runtime setup.

const searchTextAtom = Atom.make("");

effect-query

Query atoms do, because remote state needs a shared cache home.

const queryRuntime = makeRuntime(DemoApiLive);

const taskAtomFactory = createQueryAtomFactory({
  runtime: queryRuntime,
  queryKey: (taskId: string) => ["task", taskId],
  reactivityKeys: (taskId: string) => ({
    task: [taskId],
  }),
  queryFn: (taskId: string) =>
    DemoApi.use((api) => api.getTask(taskId)),
});

What to reach for in practice

  • Keep plain atoms for local state, derived state, and local writes.
  • Use a query atom when the value should be fetched and cached.
  • Use a query atom factory when that remote read is parameterized.
  • Use a mutation when a remote write should invalidate matching query atoms.

If you only remember one rule, make it this one: when the problem is "what does the app know right now," use atoms. When the problem is "how do we keep a remote value fresh and coherent over time," use query atoms.
