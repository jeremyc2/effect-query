effect-query for TanStack Query developers

If TanStack Query already makes sense to you, the hard part here is not server state. The hard part is that the center of the API moved from a client-plus-hook model to atoms plus an Effect runtime.

The two terms worth learning early are query atom and query atom factory. A query atom is the value your UI reads. A query atom factory is the reusable definition that creates query atoms from arguments.

Start with one read: the query moves out of the component

The first difference is right on the surface. In TanStack Query, you usually build the query inside the component. In effect-query, you create the query atom first and read it from the UI later.

That sounds small. It is not. It means the reactive value is already a first-class object before React touches it.

TanStack Query

The query client lives beside your hooks and gets passed to a provider.

import { QueryClient, useQuery } from "@tanstack/react-query";

const queryClient = new QueryClient();

function useDashboard() {
  return useQuery({
    queryKey: ["dashboard"],
    queryFn: fetchDashboard,
    staleTime: 60_000,
  });
}

effect-query

The query runtime also owns shared cache behavior, but it runs Effects too.

import { createQueryAtom, makeRuntime } from "effect-query";

const queryRuntime = makeRuntime();

const dashboardAtom = createQueryAtom({
  runtime: queryRuntime,
  queryKey: ["dashboard"],
  staleTime: "1 minute",
  queryFn: fetchDashboard,
});
Quick mental model: a QueryClient and a query runtime are not exact synonyms, but they do overlap. Both give you a shared cache. The extra twist is that the query runtime also executes Effects, so it can carry Effect layers and services when your queries need them.

Parameterized queries stop being hooks and become factories

This is the translation worth remembering. What used to be "a hook that takes arguments" becomes a query atom factory. Call it with an argument, and you get a query atom.

If you are looking for the one mental substitution that makes the rest of the library click, it is this one.

TanStack Query

A hook with params.

function useUser(userId: string) {
  return useQuery({
    queryKey: ["user", userId],
    queryFn: () => fetchUser(userId),
    staleTime: 60_000,
  });
}

effect-query

A factory that returns atoms.

import { createQueryAtomFactory } from "effect-query";

const userAtomFactory = createQueryAtomFactory({
  runtime: queryRuntime,
  queryKey: (userId: string) => ["user", userId],
  staleTime: "1 minute",
  queryFn: (userId: string) => fetchUser(userId),
});

const userAtom = userAtomFactory("42");

TanStack Query

The component builds the query as it reads it.

function UserScreen({ userId }: Props) {
  const user = useQuery({
    queryKey: ["user", userId],
    queryFn: () => fetchUser(userId),
  });

  if (user.isPending) return <Spinner />;
  if (user.isError) return <ErrorView error={user.error} />;

  return <UserCard user={user.data} />;
}

effect-query

The component reads an atom that already knows how to fetch.

import * as Cause from "effect/Cause";

function UserScreen({ userId }: Props) {
  const user = useAtomValue(userAtomFactory(userId));

  if (user.isPending) return <Spinner />;
  if (user.isError) {
    return <ErrorView error={Cause.pretty(user.failureCause)} />;
  }
  if (user.data !== undefined) {
    return <UserCard user={user.data} />;
  }

  return null;
}
Another important shift: in TanStack Query, the queryFn returns a Promise. In effect-query, the query returns an Effect. Same job. Different execution model. Query functions also receive a context object with queryKey and signal. The result object you read from the atom also carries status flags directly, so you do not need extra helper imports just to inspect state.

Query function context is familiar, but the execution model is not

If you are coming from TanStack Query, this should feel familiar. The query function receives the argument you passed to the factory and a context object containing the resolved queryKey and an AbortSignal.

The real difference is what you return. Promise-returning request logic becomes Effect-returning request logic, which means retries, services, and interruption stay in the same programming model as the rest of your app.

TanStack Query

Query function context.

useQuery({
  queryKey: ["user", userId],
  queryFn: ({ queryKey, signal }) =>
    fetch(`/api/users/${queryKey[1]}`, { signal }).then((r) => r.json()),
});

effect-query

The factory argument and context are both available.

const userAtomFactory = createQueryAtomFactory({
  runtime: queryRuntime,
  queryKey: (userId: string) => ["user", userId],
  queryFn: (userId, { signal }) =>
    HttpClient.get(`/api/users/${userId}`, { signal }),
});

Writes are where the vocabulary starts paying rent

TanStack Query invalidates by matching query keys. effect-query uses reactivity keys instead. You declare them once on the reads, then mutations can invalidate by intent instead of by repeating cache key shapes.

This is one of the clearest design choices in the library. Query keys still identify entries, but reactivity keys describe which entries should move together when a write lands.

TanStack Query

Invalidate by query key.

const mutation = useMutation({
  mutationFn: updateUser,
  onSuccess: (_updatedUser, variables) => {
    queryClient.invalidateQueries({
      queryKey: ["user", variables.userId],
    });
    queryClient.invalidateQueries({
      queryKey: ["users"],
    });
  },
});

effect-query

Invalidate by reactivity key.

const usersAtom = createQueryAtom({
  runtime: queryRuntime,
  queryKey: ["users"],
  reactivityKeys: () => ({
    users: ["all"],
  }),
  queryFn: fetchUsers,
});

const userAtomFactory = createQueryAtomFactory({
  runtime: queryRuntime,
  queryKey: (userId: string) => ["user", userId],
  reactivityKeys: (userId: string) => ({
    user: [userId],
  }),
  queryFn: fetchUser,
});

const updateUserMutation = createMutationAtom({
  runtime: queryRuntime,
  mutationFn: updateUser,
  invalidate: (input) => ({
    users: ["all"],
    user: [input.userId],
  }),
});

TanStack Query

Optimistic writes go through the client.

await queryClient.setQueryData(
  ["task", taskId],
  (current) => optimisticTask(current),
);

effect-query

Optimistic writes go straight to the atom factory.

await Effect.runPromise(
  taskAtomFactory.setData(taskId, (current) => {
    const base = Option.getOrElse(current, () => emptyTask(taskId));
    return optimisticTask(base);
  }),
);
The trade-off: reactivity keys are more opinionated than TanStack Query filters. The upside is that invalidation rules get declared once on the reads instead of reconstructed at every mutation site.

You still get the imperative tools, they just hang off the factory

If you liked prefetching, peeking, and hydration before navigation or during app startup, you still get those tools. They just hang off the query atom or query atom factory instead of a central client.

So the imperative escape hatch is still there. It is just closer to the query entry you actually mean to touch.

TanStack Query

Client methods.

await queryClient.prefetchQuery({
  queryKey: ["task", taskId],
  queryFn: () => fetchTask(taskId),
});

const cachedTask = queryClient.getQueryData(["task", taskId]);

effect-query

Factory and atom methods.

await Effect.runPromise(taskAtomFactory.ensure(taskId));

const cachedTask = await Effect.runPromise(taskAtomFactory.peek(taskId));
await Effect.runPromise(taskAtomFactory.cancel(taskId));

TanStack Query

Hydrate the client.

import { dehydrate, hydrate } from "@tanstack/react-query";

const snapshot = dehydrate(queryClient);
hydrate(queryClient, snapshot);

effect-query

Hydrate the registry that owns the atoms.

import * as AtomRegistry from "effect/unstable/reactivity/AtomRegistry";
import { dehydrate, hydrate } from "effect-query";

const sourceRegistry = AtomRegistry.make();
const targetRegistry = AtomRegistry.make();

const snapshot = dehydrate(sourceRegistry);
hydrate(targetRegistry, snapshot);

Freshness, polling, and offline behavior are three separate knobs

Just like in TanStack Query, freshness and polling are not the same setting. Use staleTime to define freshness, refetchInterval to poll, and networkMode to control how queries behave while offline. 

const activityAtom = createQueryAtom({
  runtime: queryRuntime,
  queryKey: ["activity"],
  staleTime: "30 seconds",
  refetchInterval: "5 seconds",
  networkMode: "online",
  queryFn: fetchActivity,
});

If a query should stay fresh for 30 seconds, that does not imply it should poll every 30 seconds. And if it should poll, that still does not answer what should happen while the app is offline.

What changes in practice

  • useQuery(...) becomes createQueryAtom(...) or createQueryAtomFactory(...), then your UI reads the resulting atom.
  • A query runtime is the shared home for cache behavior, and it also runs Effects.
  • A parameterized query is a query atom factory. Calling it returns a query atom.
  • Invalidation moves from query-key matching to reactivity-key matching.

If you already know TanStack Query, the shortest accurate summary is: same class of problems, different center of gravity. The queries are still about cache identity, freshness, invalidation, and hydration. They just live in atoms and Effects instead of observers and promises.
